# FLEET LOOP MASTER DIRECTIVE

# JobHunterAI — Career Intelligence Platform

## V10 Executive Operating Directive

---

## Executive Mission

Your mission is to evolve JobHunterAI into a **production-grade Career Intelligence Platform** that measurably improves the candidate's probability of obtaining a high-quality **paid remote software engineering, backend, AI, or analytics position**.

The platform is not judged by code written.
The platform is judged by measurable career outcomes.

Primary KPIs:
- Recruiter Response Rate
- Interview Rate
- Assessment Rate
- Offer Rate
- Offer Quality
- Candidate Productivity

Everything else is a supporting metric.

---

## North Star

Every engineering decision must answer one question:
> **Will this improve the candidate's probability of receiving a better interview or offer?**

If the answer cannot be justified with evidence or reasoning, deprioritize the work.

---

## Product Definition

JobHunterAI is NOT:
- a resume generator
- a cover letter generator
- an auto-apply bot
- an ATS checker

JobHunterAI IS:
- Career Intelligence Platform
- Recruitment Operating System
- Knowledge-Driven Career Assistant
- Career Analytics Platform
- Opportunity Intelligence Engine

---

## Operational Philosophy

The system must continuously:
```
Discover ➔ Analyze ➔ Rank ➔ Research ➔ Generate ➔ Validate ➔ Explain ➔ Present ➔ Track ➔ Learn ➔ Improve
```
This cycle repeats continuously until the candidate manually accepts an offer or pauses the workflow.

---

## Product Objectives

The platform shall:
- discover legitimate remote opportunities
- identify high-value companies
- tailor evidence-backed resumes
- generate truthful cover letters
- prepare recruiter outreach
- prepare interview material
- track recruiter interactions
- measure application outcomes
- identify skill gaps
- recommend career improvements
- generate daily and weekly reports

---

## Candidate Optimization

Always optimize toward:
- Backend Engineering
- AI Engineering
- Software Engineering
- Analytics Engineering
- Data Engineering
- Remote-first companies
- Developer tooling
- AI infrastructure
- SaaS
- High-growth startups
- Global remote employers

Never optimize toward irrelevant opportunities simply to increase application count.

---

## OMI Strategy

Treat OMI as the flagship engineering project.
Present different implemented capabilities depending on role:
- **Backend**: FastAPI, API Design, Infrastructure, Docker, PostgreSQL
- **AI**: Routing, Evaluation, LLM Orchestration
- **Analytics**: Telemetry, Metrics, Evaluation Pipelines

Only present completed, verifiable capabilities.
Never represent roadmap items as finished work.

---

## Resume Intelligence

Resumes shall never be manually assembled.

Pipeline:
```
Job ➔ Knowledge Graph ➔ Evidence Selection ➔ Role-Based Ranking ➔ Resume Compilation ➔ ATS Optimization ➔ Truth Validation ➔ Human Review ➔ Output
```
Every bullet must be supported by evidence.

---

## ATS Philosophy

ATS optimization is a constraint—not the objective.

Prioritize:
- truthful keyword alignment
- relevance
- readability
- clean formatting
- project quality

Do not chase arbitrary ATS percentages.
Stop optimization when no further truthful improvement exists.

---

## Knowledge Graph

The knowledge graph is the source of truth.
```
Evidence ➔ Skill ➔ Project ➔ Experience ➔ Resume Bullet ➔ Resume Version ➔ Application ➔ Interview ➔ Offer
```
Every generated statement must be explainable.

---

## Deterministic vs AI

- **Deterministic**: ranking, validation, scoring, scheduling, graph traversal, analytics, caching, business rules.
- **LLM**: language generation, summarization, cover letters, outreach, interview preparation.

Never replace deterministic logic with AI.

---

## Discovery Strategy

Continuously discover opportunities from supported sources.
Normalize.
Deduplicate.
Score.
Reject weak opportunities.
Prioritize quality over quantity.

---

## Human Approval

The platform prepares.
The candidate decides.

Never automatically submit applications.
Always require explicit approval before external actions.

---

## Analytics

Track:
```
Applications ➔ Responses ➔ Assessments ➔ Interviews ➔ Offers ➔ Acceptance
```
Measure:
- interview conversion
- resume performance
- recruiter responsiveness
- project effectiveness
- outreach effectiveness

---

## Learning

Use historical outcomes to improve:
- project ranking
- resume ordering
- keyword emphasis
- outreach strategy
- interview preparation

Never reinforce unsupported claims.

---

## Explainability

Every decision must answer:
- Why was this project selected?
- Why was another rejected?
- Why did ATS improve?
- Which evidence supports this bullet?
- What changed compared to the previous version?

Every recommendation must be auditable.

---

## Deployment

Cloud-first architecture.
The platform must not depend on localhost.

Support:
- secure authentication
- background workers
- scheduled jobs
- monitoring
- logging
- backups
- scalable deployment

---

## UI Philosophy

The interface should feel like a premium SaaS product.

Visual direction:
- Claymorphism
- Brutalist layouts

Characteristics:
- responsive
- accessible
- fast
- keyboard friendly
- minimal
- elegant

Preferred ecosystem (preferred tools, not mandatory constraints):
- React, Next.js, TypeScript, Tailwind
- Firebase, Figma, Lucide, AutoAnimate, Storylane, CodeRabbit

Engineering quality takes precedence over specific tooling.

---

## Engineering Workflow

Every feature follows:
```
Research ➔ Plan ➔ Implement ➔ Static Analysis ➔ Type Checking ➔ Unit Tests ➔ Integration Tests ➔ Accessibility Review ➔ Performance Review ➔ Security Review ➔ Automated Code Review ➔ Documentation ➔ Deployment Validation ➔ Human Review ➔ Merge
```
Never bypass quality gates.

---

## Validation Standard

Do not assume correctness.

Before declaring work complete:
- verify implementation
- verify tests
- verify documentation
- verify performance
- verify security
- verify accessibility
- verify maintainability
- verify production readiness

Where automated verification is insufficient, clearly identify what still requires manual validation rather than claiming certainty.

---

## Continuous Refactoring

After every development cycle:
- remove duplication
- simplify architecture
- improve modularity
- improve documentation
- improve testing
- reduce technical debt

Favor long-term maintainability over short-term feature growth.

---

## Constraints

Never:
- fabricate qualifications
- fabricate evidence
- fabricate metrics
- misrepresent open-source work
- bypass validation
- violate platform terms of service
- claim successful real-world outcomes without supporting data
- automatically submit applications on behalf of the candidate

Truthfulness is non-negotiable.

---

## Completion Criteria

A task is complete only when:
- [x] It improves the platform's objectives.
- [x] It passes all quality gates.
- [x] Documentation is updated.
- [x] Tests pass.
- [x] The implementation is maintainable.
- [x] The change is explainable.
- [x] The feature provides measurable value.

If any criterion fails, continue refinement before considering the work complete.

---

## Final Directive

From this point onward, prioritize **execution over architecture**.

Do not add new systems merely because they are technically interesting.
Build only what measurably improves the candidate's ability to discover better opportunities, submit stronger applications, prepare more effectively for interviews, and make better career decisions.

Every commit should move the platform closer to becoming a reliable, production-ready Career Intelligence Platform rather than increasing architectural complexity alone.
